## Getting Started
To get started with Strikes, you'll need to install the `dreadnode` package, which is the Python SDK that powers Strikes functionality. 

<CodeGroup>
```bash pip
pip install -U dreadnode
```

```bash uv
uv pip install dreadnode
```

```bash uv (project)
uv add dreadnode
```

```bash poetry (project)
poetry self add dreadnode
```
</CodeGroup>


### Authentication
After installing the package, you'll need to authenticate with the Dreadnode Platform. This can be done using the command line interface (CLI) or by configuring your environment variables.

<CodeGroup>
```bash cli
# Authenticate to platform.dreadnode.io
dreadnode login

# For self-hosted platforms, specify the server URL
dreadnode login --server http://self-hosted
```

```bash env
# Set environment variables for authentication
export DREADNODE_API_KEY="your_api_key"
export DREADNODE_SERVER_URL="server_url"
```

</CodeGroup>

<Note>
For complete authentication and configuration guidance, see the [Configuration](/usage/config) documentation.
</Note>

### Logging a Run
If you've used an experiment tracking library before, this code will be familiar. All the functions you're familiar with work exactly as you would expect. Here are the most common functions you'll use:

- `log_param()`: Log hyperparameters, target models, or agent configs.
- `log_metric()`: Log measurements you take anywhere in your code.
- `log_input()`: Log runtime objects like prompts, datasets, samples, or target resources.
- `log_output()`: Log runtime objects like findings, commands, reports, or model outputs.
- `log_artifact()`: Log local files or directories like source code, container configs, datasets, or models.

You can call `dreadnode.log_*` functions anywhere in your code, and you don't have to worry about keeping track of your active run or task. Everything just works.

```python basic example
import asyncio
import dreadnode

NAMES = ["Bob", "Alice", "Sarah", "Jeff"]

# Create a new task
@dreadnode.task()
async def say_hello(name: str) -> str:
    return f"Hello, {name}!"

async def main() -> None:

    # Start a new run
    with dreadnode.run("first-run"):
        # Log parameters
        dreadnode.log_params(
            name_count=len(NAMES),
        )

        # Log inputs
        dreadnode.log_input("names", NAMES)

        # Run your tasks
        for name in NAMES:
            # Call the task
            greetings = await say_hello(name)

            # Log the output
            dreadnode.log_output(f"greeting_{name}", greetings)

            # Log a metric
            dreadnode.log_metric("greeting_length", len(name))

        # Save the current script
        dreadnode.log_artifact(__file__)

asyncio.run(main())
```

This example code did a few things:

1. Created a new "Default" project in the Platform to hold the run
2. Began an OpenTelemetry trace for all code under `with dreadnode.run(...)`
3. Tracked and stored our `log_*` calls and the tracing information.
4. Saved the current script as an artifact in the run.
5. Sent the data to the [Platform](https://platform.dreadnode.io/strikes/projects/Default).


## Core Concepts
Strikes is built around three core concepts: **Runs**, **Tasks**, and **Metrics**. You functionally organize your code into runs, which contain tasks that perform units work. Metrics are used to measure and evaluate the performance of your runs and tasks.

### Runs
Runs are the context for execution in Strikes, they are the place where all your data is collected and represent a complete execution session. Think of runs as an single "experiment" or "session" for your code.

```python
import dreadnode

with dreadnode.run("my-experiment"):
    # Everything that happens here is part of the run
    # All data collected is associated with this run
```

You can create multiple runs, even in parallel, to organize your work logically:

```python
async def work(target: str):
    with dreadnode.run(target):
        # Run-specific work here
        pass

await asyncio.gather(*[work(f"target-{i}") for i in range(3)])
```

See the [Runs](/usage/runs) page for more details on creating, configuring and managing runs.


### Tasks

Tasks are units of work within runs. They help you structure your code and provide a finer-grained context for data collection. Tasks can be created as function decorators or context managers:

```python
import dreadnode

@dreadnode.task()
async def say_hello(name: str) -> str:
    return f"Hello, {name}!"

with dreadnode.run():
    with dreadnode.task_span("manual-task"):
        # Task work here
        pass

    # Call the decorated task
    result = await say_hello("Alice")
```

Tasks can automatically track their inputs, outputs, execution time, and more. They form the foundation for building structured, observable workflows.

See the [Tasks](/usage/tasks) page for more details on task creation, configuration, and advanced patterns.

### Metrics

Metrics are attached to tasks, or called manually. They allow you to evaluate the effectiveness of your agents and track important events during execution:

```python
import dreadnode

@dreadnode.task
async def take_action(input: str) -> str:
    # ...

    dreadnode.log_metric("num_valid_actions", 1, mode="count")

with dreadnode.run():
    # Log a simple metric
    dreadnode.log_metric("accuracy", 0.87)

    # Log a metric with a step number for timeseries data
    for step in range(10):
        dreadnode.log_metric("loss", 0.23, step=step)
```

Metrics can be associated with tasks, runs, or even specific objects in your system, providing a comprehensive view of performance at different levels.

See the [Metrics](/usage/metrics) page for more information on creating, aggregating, and analyzing metrics.
